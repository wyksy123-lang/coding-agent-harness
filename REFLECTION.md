\# 项目反思



\## 1. 项目与个人职责



本次 AI4SE 期末项目中，我选择完成 Coding Agent Harness。项目的核心目标是围绕编码任务构建一套可控制、可测试、可追踪的智能体运行框架。系统需要接收自然语言需求，让模型选择工具，并通过文件读写、命令执行、测试运行、失败分类、反馈注入、记忆检索以及人工审批等机制形成完整闭环。



在项目中，我既是需求提出者，也是项目管理者、架构设计者和最终审核者。大部分代码由 Codex、OpenCode 等编码智能体生成，但任务划分、功能取舍、错误判断、分支管理、CI 配置、部署验证和最终是否接受代码，都需要由我负责。



我最初以为，只要把完整需求交给智能体，它就能够自动完成项目。但实际开发中，遇到了很多问题，比较重点的是我一开始遵循ai的提醒将opencode下载在了ubuntu中，后续在api金额耗尽后我又将剩余任务交由windows系统中的codex完成，其中产生了诸多兼容性问题。在prompt方面，我实际在运行到task10时我的金额就已经耗完，这使我反思为什么token用量这么大，后续发现我的设定下每次task都会将包括plan，spec等文档全部重新当作prompt输入，这种重复导致了大量的token浪费，于是我重新追加了命令，简化了prompt，最终取得了很好的效果。我花费相当多精力判断agent当前位于哪个分支、是否错误继承了旧任务、测试失败到底来自实现还是环境、某个结果能否作为真实完成证据，以及是否应当继续下一项任务。我需要合理规定agent的输入输出，让它能全面反馈每一次task的运行结果，当然这个过程是项目中最痛苦的过程，目前这个项目需要的很多指标我并不理解，所以需要将输出结果转交给其他大模型给我解释，然后我再去给agent下合理的命令去修复。



例如，在 T13 到 T28 的推进过程中，我需要反复处理 worktree、分支、PR、Windows 环境、Python 版本、代理连接、GitHub Actions 和 Render 部署等问题。这让我认识到如今的aicoding虽然火爆，但若没有提前指定严格的审查流程，其安全性和功能是没有办法保证的，按task创建worktree并push和merge，可以最大程度保证已完成部分的稳定。

再比如我在执行merge后发现已经完成的 Task13 仍出现在 worktree list 中，询问chatgpt发现实际删除时提示 worktree 正在使用、目录无法删除或分支被 worktree 占用，后续还需要关闭 Codex、VS Code、PowerShell，执行 git worktree remove、git worktree prune，有时还涉及管理员 PowerShell和 Windows 文件占用，这些都极大扩充了我对git开发过程的认知。



\## 2. 最有价值的 Superpowers 技能



在使用的 Superpowers 技能中，我认为最有价值的是 `writing-plans`、`test-driven-development`、`systematic-debugging` 和 `verification-before-completion`。



`writing-plans` 的价值在于把一个模糊的大任务拆成较小、具有依赖关系的任务。我的项目最终被划分为配置、LLM 抽象、工具、治理、反馈、记忆、主循环、CLI、WebUI、CI、分发和部署等多个部分。如果没有 PLAN，智能体很容易在一次执行中同时改动多个模块，使错误来源难以定位。



`test-driven-development` 保证了每个任务都有 Red、Green 和 Refactor 阶段。它让我能够通过提交历史证明测试先于实现存在，而不是在代码完成后补几个测试作为装饰。例如每个任务通常包含：



\- Red：先增加失败测试；

\- Green：完成最低可行实现；

\- Refactor/Review：修复评审问题；

\- Docs：记录任务证据。



这种历史不仅服务于作业检查，也使我能够在后续出现问题时定位变化来自哪个阶段。



`systematic-debugging` 对 Windows 兼容问题尤其有价值。项目曾出现 `os.O\_NOFOLLOW`、符号链接权限、临时目录权限、`python3`、`pwd` 和 `sleep` 等跨平台问题。如果只是看到报错就逐个打补丁，很可能破坏 Linux 行为。通过先分类问题，我将它们区分为 Windows 不兼容、环境依赖缺失、测试设计错误和真实实现缺陷，再分别处理。



`verification-before-completion` 则改变了我对“完成”的判断。智能体经常会说“任务完成”，但如果没有实际运行完整测试、Ruff、Mypy、构建和远程 CI，就不能认为任务真的完成。后期我逐渐养成了先看命令结果、退出码和远程状态，再接受结论的习惯。



\## 3. 哪些流程形式大于实质



虽然 Superpowers 的流程很完整，但并不是所有步骤在本项目中都同样有效。其中一部分流程容易出现形式大于实质的问题。



首先，每个任务都要求完整经历 Red、Green、Refactor、Docs，并记录多个评审者。在较大的核心任务中，这种流程很有价值；但在只修改一份配置文件或 README 的任务中，严格制造一个失败测试有时显得机械。例如 README 的 Red 阶段只是验证文件或章节尚不存在，这能够满足 TDD 的形式，却不能真正说明文档质量。



其次，部分“subagent 评审”有时只是输出一段格式规范的总结。如果没有人工查看代码差异、测试范围和风险，评审记录再完整也不代表代码真的可靠。我发现智能体经常倾向于给出“无 Critical 问题”的结论，但后续仍可能在 Windows、打包或远程 CI 中暴露问题。



再次，AGENT\_LOG 和需求清单随着任务数量增加变得非常庞大。如果每次都重复记录相似命令和说明，文档容易变成日志堆积，而不是帮助读者理解关键决策。后期我更倾向于只记录重要偏离、重大错误、人工决策和可复现证据。



因此，我认为流程本身不能代替判断。Red、Green、Review 等标签只有与真实命令、真实失败和真实修改对应时才有意义。



\## 4. TDD 



我认为 TDD 在本项目中既是短期阻碍，也是长期放大器。



它的阻碍主要体现在开发速度上。对于一个很简单的功能，先写失败测试、提交、再实现、再重构，会比直接完成代码花费更多时间和 Token。智能体有时还会为了制造 Red 阶段，编写过度依赖内部实现的测试，导致后续重构困难。



但是，当项目规模扩大后，TDD 的收益逐渐明显。Harness 涉及文件系统、命令执行、测试报告解析、失败分类、记忆存储、HITL 状态和 AgentLoop 等多个模块。如果没有确定性测试，任何一个修改都可能影响其他部分。



例如 Windows 兼容修复最初只针对几个失败，但完整测试暴露出临时目录、符号链接和 shell 命令的连锁问题。通过单元测试和回归测试，我能够确认修复没有破坏之前已完成的工具与反馈机制。



TDD 最大的价值不是证明代码永远正确，而是为 AI 生成代码提供一个客观反馈信号。模型可能对需求产生误解，但测试结果不会因为模型的解释而改变。对 Coding Agent Harness 来说，这一点尤其重要，因为“测试—分类—反馈—再执行”本身就是项目的核心机制。



\## 5. Subagent 工作流的有效自主运行边界



从本项目经验看，Subagent 最适合处理边界清晰、输入完整、验证命令明确的任务。例如新增一个类、实现一个解析器、增加一组测试、修改一个 CI job，这些任务通常可以由智能体独立完成。



当任务同时涉及多个外部系统或隐含状态时，智能体的自主运行能力会明显下降。例如：



\- 当前工作区与分支不一致；

\- 任务依赖的 PR 尚未合并；

\- 本地 main 未同步；

\- Codex worktree 指向旧任务；

\- Windows 和 Codex 使用了不同 Python；

\- GitHub、Render 和本地环境状态不一致；

\- 远程服务需要登录和人工授权。



这些问题无法只靠阅读代码解决。智能体可能看到某个 commit 存在，就误以为远程已经合并；也可能在旧分支上完成新任务，导致后续提交关系混乱。



因此，我认为 Subagent 的合理边界是：它可以自主完成局部实现和验证，但不应独立决定跨任务状态、远程部署凭据、分支合并和最终验收。项目管理层面的事实必须由人确认。



\## 6. 最合适的 Task 颗粒度



本项目中最合适的 Task 颗粒度，是一个智能体能够在一次会话中理解、实现、测试和评审，同时修改文件数量不至于过多。



例如，T12 测试结果解析、T13 失败分类、T15 反馈注入和 T16 轮次追踪分别拆开是合理的。它们共同组成反馈闭环，但每个任务都有清晰输入和输出，也能独立测试。



颗粒度过大时，例如同时要求完成 CLI、WebUI、Docker 和部署，智能体会频繁切换上下文，容易遗漏需求。颗粒度过小时，例如只为一个配置键建立独立任务，又会带来大量分支、PR、文档和 Token 开销。



根据本次经历，我认为一个任务通常适合满足以下条件：



1\. 主要目标只有一个；

2\. 修改约 2–8 个核心文件；

3\. 有一组明确的目标测试；

4\. 可以在一次会话中完成；

5\. 不依赖尚未发生的外部操作；

6\. 完成后能形成独立、可解释的 commit。



再比如另一个我头疼过的点：如前文提到的windows系统兼容的问题，我先推送并合并了 Windows 修复，再推送 Task22，但合并 Task22 时出现冲突，因为两个分支修改了相同或相邻文件。两个分支都基于较旧的 main 开始，导致后合并的分支必须重新解决冲突。这说明如果要重做之前任务，我会在开始新任务前先同步最新 main；对于会修改公共测试基础设施的任务，尽量串行执行，而不是并行创建多个长期分支。



\## 7. SPEC/PLAN 不清导致偏离的具体案例



项目中比较明显的一次偏离发生在 Windows 兼容性上。早期 SPEC 和 PLAN 更关注 Harness 的功能机制，没有明确将 Windows 11 + PowerShell 设为一级运行平台。部分实现默认了 Linux/POSIX 环境，例如使用 `os.O\_NOFOLLOW`、`python3`、`pwd`、`sleep` 和符号链接能力。



这些实现单独看并不错误，但与我的实际开发环境不一致，导致完整测试中出现大量失败。起初智能体把它们统称为“已有 Windows baseline”，倾向于忽略；但如果最终目标是让其他 Windows 用户也能运行，就不能把这些失败当作正常现象。



后来我增加了 FIX-WIN-01，明确：



\- Windows 11 + PowerShell + Python 3.11 是主要环境；

\- Linux 和 Docker 保持兼容；

\- Windows 全量 pytest、Ruff 和 Mypy 不能保留已知失败；

\- 测试应探测真实能力，而不是仅根据 API 是否存在来跳过。



这个案例让我认识到，SPEC 中未写明的平台假设会直接变成实现偏差。智能体不会自动知道“我主要在 Windows 使用”意味着所有 shell、路径、临时目录和进程控制都必须重新审视。

比如最初的PLAN.md把 make test 当成统一验收命令，但没有考虑 Windows 默认不存在 Make。这里真正重要的是“一键、可重复地运行全部检查”，而不是命令名称本身。我在 Windows 使用等价的 Python 模块命令，在 GitLab Linux Runner 中保留真实 Make 验证。这让我认识到，流程需要守住目标，而不应机械坚持某个工具形式。

同时我还遇到一个问题：我的电脑上同时安装了 Anaconda 和 Python，造成了实际运行的解释器可能有：Anaconda Python 3.13.5；项目要求的 Python 3.11；可能还有 Codex 自带或其他路径中的 Python；.venv 中的项目解释器。这导致：Codex 执行的 python 和 PowerShell 中的 python 不是同一个；某些包在一个环境已安装，在另一个环境却缺失；pytest、Mypy、Ruff 结果不一致。最后的解决方案是创建了conda环境harness311，并在仓库内创建.venv\\Scripts\\python.exe，要求codex以后显式使用该解释器。



\## 8. 最有效的 Prompt 与 Context 策略



本项目中最有效的 Prompt 不是最短的 Prompt，而是能够明确状态、边界、验证和停止条件的 Prompt。



后期我使用的 Prompt 通常包含：



\- 当前任务编号和分支；

\- 只允许完成哪一个任务；

\- 禁止开始下一任务；

\- 必须读取哪些文档；

\- 不需要重复读取哪些内容；

\- 必须使用哪个 Python 解释器；

\- 需要运行哪些测试；

\- 哪些失败不能忽略；

\- 不得 push 或 merge；

\- 最终输出哪些证据；

\- 在什么条件下必须停止询问。



例如在 T26 中，如果只说“完成 Docker 和 PyPI 打包”，智能体可能使用系统 Python 3.13，或者只生成 Dockerfile 而不验证 wheel。后来 Prompt 明确要求使用仓库中的 Python 3.11 `.venv`、从仓库外安装 wheel、检查静态资源、验证 `harness --help` 和 WebUI HTTP 200，任务结果才具有可移植性证据。



我还发现，将历史对话全部交给智能体不一定有效。过多上下文会增加 Token 消耗，也可能让它继承旧任务状态。更好的方式是提供当前任务所需的最小上下文，并让仓库中的 SPEC、PLAN、AGENTS 和 CODEX\_MODE 成为稳定事实来源。



\## 9. 凭据治理和分发带来的新认识



在这个项目之前，我更关注“代码是否能运行”，较少考虑其他用户如何安全地获取、安装和配置项目。凭据和分发要求迫使我补上了这一部分工程思维。



凭据方面，API Key 不能硬编码、不能提交到 Git、不能打印到日志，也不能直接写在命令行中。项目实现了 CredentialManager，优先使用系统 Keyring，并提供 setup、status、update 和 clear 操作。即使回退到 `.env`，也必须说明它是明文文件并加入 `.gitignore`。



我认识到“没有把 Key 写进源码”并不等于安全。Key 仍可能出现在 Git 历史、终端历史、CI 日志、Docker layer、README 示例或错误输出中。因此最终验收需要同时扫描当前工作树和提交历史。



分发方面，项目不仅需要在我的开发目录中运行，还需要构建 wheel、sdist 和 Docker 镜像，并在全新虚拟环境中安装。T26 的 fresh-environment 验证让我看到，源码目录中的隐式导入、未打包静态文件和本机已有依赖，都可能制造“在我电脑上正常”的假象。



Render 部署则进一步验证了 Dockerfile、端口、静态资源和公网访问是否真正工作。这些环节让我从“实现功能”转向思考“交付一个别人能够运行的系统”。



\## 10. 对 Superpowers 的思考



Superpowers 的核心假设是：软件任务可以被拆分为清晰的规格、计划和小任务，并且通过 TDD、评审和分支隔离降低智能体偏离风险。这一假设在模块化实现中基本成立，但在外部状态复杂的任务中并不完全成立。



第一，它假设需求可以在实现前充分明确。但实际项目中，Windows 兼容、Python 版本、Render 端口和 GitHub CI 等问题，往往只有运行后才暴露。SPEC 不可能一次性覆盖所有环境差异，因此流程必须允许基于证据持续修正规约。



第二，它假设更多评审能提升质量，但 AI 评审者可能共享相似盲点。两个 Subagent 都认为代码没有问题，并不代表远程 CI 或真实 Windows 环境不会失败。独立环境的执行证据往往比更多文字评审更可靠。



第三，它带来了较高的流程成本。大量小任务意味着大量 worktree、分支、PR、日志和上下文切换。对于个人课程项目，这种成本有时超过代码本身。



第四，它容易让人把“完成流程”误认为“完成工程”。真正重要的仍然是：系统能否安装、测试是否真实通过、凭据是否安全、用户能否运行以及开发者能否解释关键决策。



因此，我认为 Superpowers 更适合作为约束智能体的脚手架，而不是不可调整的流程标准。工程师必须知道什么时候严格遵守，什么时候根据项目风险进行裁剪，并对偏离负责。

比如PLAN.md把 make test 当成统一验收命令，但没有考虑 Windows 默认不存在 Make。这里真正重要的是“一键、可重复地运行全部检查”，而不是命令名称本身。最终我在 Windows 使用等价的 Python 模块命令，在 GitLab Linux Runner 中保留真实 Make 验证。



\## 11. 总结

虽然完成了项目，但是我个人并不满意：

这个harness只是实现了满足项目要求的最基本功能，实际上只能通过命令行来操作，没有如codex那样清晰的操作界面，而且我个人是很希望有个web界面来操纵harness，但是这又涉及到整个项目的重构，需要从头开始修改，于是放弃。通过这个项目，我基本了解了设计harness的原理，参与了git介入的开发流程，以及改变了对aicoding产品的看法。aicoding在合理的流程下才能创造供他人使用的产品，为未来时代下的软件工程创造了可能的需求：既然ai已经将写代码的能力赋予大众，那软件工程可以转向帮助用户确定自己的aicoding产品是否安全可行。

我不禁开始幻想，如果重新开始这个项目，我会怎么做：



第一，在 SPEC 初期就明确目标平台和支持矩阵，将 Windows 11、PowerShell、Python 3.11、Ubuntu CI 和 Docker 写成可验收要求，而不是后期通过修复任务补充。



第二，我会更早建立统一的本地虚拟环境，并要求所有智能体显式使用同一个解释器，避免 Anaconda Python、系统 Python 和 Codex bundled Python 混用。



第三，我会减少任务间重复文档，把详细命令输出保存为结构化验收记录，AGENT\_LOG 只保留关键决策、人工干预和教训。



第四，我会在项目早期增加 Docker Build 和 fresh-wheel-install CI，而不是等到最终阶段才验证可移植性。



第五，我会减少过细的任务和重复评审，将紧密相关、风险较低的配置修改合并，但仍保留核心机制的独立 TDD 提交。



第六，我会更早设计 CLI 与 WebUI 的关系。目前 WebUI 主要是状态展示和 HITL 界面，并没有直接连接 CLI 中运行的 AgentLoop。如果时间充足，可以增加受控的任务管理器和事件桥接，使 WebUI 真正展示 Agent 的执行过程。但这属于增强功能，不应影响核心 Harness 的安全边界。
